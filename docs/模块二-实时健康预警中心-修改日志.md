# 智慧医养大数据公共服务平台 — 医生服务系统

## 模块二：实时健康预警中心 — 修改日志

---

## 一、变更概述

本次修改新增"实时健康预警中心"能力，覆盖预警的**产生 → 推送 → 处理**全流程，采用 SSE（Server-Sent Events）直接推送方案实现服务端主动推送，医生端无需刷新页面即可实时收到预警通知。

- **推动机制**：单机模式使用 SSE 直接推送（`WarningServiceImpl.publish()` 直接调用 `WarningSseManager.pushToUser()`）；Redis 发布/订阅作为分布式部署的备选通道（默认禁用，通过 `medical.warning.redis-pubsub.enabled=true` 开启）。
- **负责范围**：仅负责预警的实时生成、推送与处理闭环。
- **不涉及范围**：不涉及 AI 健康评估报告、不涉及重点人群风险分层（由其他成员负责的分支完成）。

## 二、数据库变更（`src/main/resources/db/schema.sql`）

### 1. `health_warning` 表新增字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `source_data_id` | BIGINT | 触发该预警的体征数据ID |
| `push_status` | TINYINT | 推送状态：0未推送 1已推送 2推送失败 |
| `read_status` | TINYINT | 已读状态：0未读 1已读 |
| `handle_deadline` | DATETIME | 处理截止时间（红色2小时/橙色6小时/黄色24小时） |

新增索引 `idx_read_status`。变更采用幂等存储过程（`IF NOT EXISTS` 判断列是否存在）追加，兼容已有数据，重复执行不报错。

> **⚠ 已知问题**：Spring Boot 的 `DataSourceScriptDatabaseInitializer` 使用 `ScriptUtils` 逐条执行 SQL，不支持 MySQL 客户端专属的 `DELIMITER` 语法，导致存储过程创建失败。首次部署时需要手动执行以下 DDL 补全 `health_warning` 表结构：
> ```sql
> ALTER TABLE health_warning ADD COLUMN source_data_id BIGINT DEFAULT NULL COMMENT '触发该预警的体征数据ID' AFTER threshold_value;
> ALTER TABLE health_warning ADD COLUMN push_status TINYINT DEFAULT 0 COMMENT '推送状态';
> ALTER TABLE health_warning ADD COLUMN read_status TINYINT DEFAULT 0 COMMENT '已读状态';
> ALTER TABLE health_warning ADD COLUMN handle_deadline DATETIME DEFAULT NULL COMMENT '处理截止时间';
> ALTER TABLE health_warning ADD INDEX idx_read_status (read_status);
> ```
> 建议后续将存储过程改造为纯 `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` 语句（MySQL 8.0+ 原生支持），而不使用 `DELIMITER` + `CREATE PROCEDURE`。

### 2. 新增表 `warning_event_log`（预警生命周期事件日志）

记录预警从创建到关闭的每一次状态流转：`创建`/`推送`/`已读`/`处理`/`忽略`/`关闭`。

| 字段 | 说明 |
|------|------|
| `warning_id` | 关联预警ID |
| `event_type` | 1创建 2推送 3已读 4处理 5忽略 6关闭 |
| `operator_id` | 操作人ID |
| `event_content` | 事件说明 |
| `create_time` | 创建时间 |

### 3. 新增表 `warning_push_channel`（推送在线连接记录）

记录医生/管理员端 SSE 连接的在线状态，便于排查推送是否送达。

| 字段 | 说明 |
|------|------|
| `user_id` | 用户ID（医生/管理员） |
| `channel_type` | 1医生 2管理员 |
| `client_id` | 客户端连接标识（唯一） |
| `status` | 0离线 1在线 |

## 三、后端变更

### 新增文件

| 文件 | 作用 |
|------|------|
| `entity/WarningEventLog.java` | 预警事件日志实体 |
| `entity/WarningPushChannel.java` | 推送连接记录实体 |
| `mapper/WarningEventLogMapper.java` | 事件日志 Mapper |
| `mapper/WarningPushChannelMapper.java` | 连接记录 Mapper |
| `common/push/WarningSseManager.java` | SSE 连接池管理（按用户ID维护 `SseEmitter` 列表，超时/异常/完成时自动清理，避免内存泄漏） |
| `common/config/RedisMessageConfig.java` | Redis 发布/订阅配置（**默认禁用**），监听 `warning:created`/`warning:updated` 频道。通过 `@ConditionalOnProperty(name="medical.warning.redis-pubsub.enabled", matchIfMissing=false)` 控制，仅在分布式部署时通过配置开启。避免了 Spring Boot 自动配置的 `RedisMessageListenerContainer` 与自定义容器冲突导致 `NullPointerException` 的问题 |
| `controller/WarningStreamController.java` | `GET /api/warnings/stream`：建立 SSE 长连接，登记在线通道 |

### 修改文件

| 文件 | 变更内容 |
|------|---------|
| `entity/HealthWarning.java` | 新增 `sourceDataId`/`pushStatus`/`readStatus`/`handleDeadline` 字段 |
| `service/WarningService.java` | 新增接口方法 `createAndPush`/`markRead`/`getRealtimeStats` |
| `service/impl/WarningServiceImpl.java` | `createAndPush()` 作为预警创建统一入口：写库 → 写生命周期日志 → 计算处理截止时间 → 写时间轴 → 解析责任医生 → **SSE 直接推送**（同时通过 Redis `convertAndSend` 发送给分布式集群的其它节点）；`handle()`/`ignore()` 补充事件日志与时间轴记录；新增 `markRead()`、`getRealtimeStats()`；新增 `@Autowired WarningSseManager` 注入，`publish()` 方法在 Redis 发布之外增加 `warningSseManager.pushToUser()` 本地直接推送 |
| `service/impl/WarningRuleServiceImpl.java` | 规则命中创建预警时改为调用 `warningService.createAndPush()`，复用统一的推送与日志逻辑，避免重复写时间轴 |
| `controller/WarningController.java` | 新增 `PUT /api/warnings/{id}/read`、`GET /api/warnings/stats/realtime` 两个接口 |

### 新增/变更接口一览

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/warnings/stream` | 建立 SSE 实时推送连接（`?token=` 鉴权，供 `EventSource` 使用） |
| PUT | `/api/warnings/{id}/read` | 标记预警为已读 |
| GET | `/api/warnings/stats/realtime` | 获取未读数/高危未处理数/今日新增数 |

### 技术要点

- 使用 `StringRedisTemplate` 发布 JSON 字符串消息，避免自定义 `RedisTemplate<String,Object>` 的 Jackson 类型序列化在订阅端解析复杂化。
- `EventSource` 无法自定义请求头，鉴权复用现有 `JwtInterceptor` 已支持的 `?token=` 查询参数兜底方案，无需改动拦截器。
- `SseEmitter` 设置为不超时（由前端断线重连控制），`onCompletion`/`onTimeout`/`onError` 回调统一清理连接，防止内存泄漏。

## 四、前端变更（`src/main/resources/static/pages/{index.html, app.js}`）

### `app.js`

- `data()` 新增：`warningStream`（EventSource 实例）、`warningReconnectTimer`、`realtimeStats: { unread, high, todayNew }`、`warningPopup`（当前高危弹窗预警对象）。
- 新增方法：
  - `connectWarningStream()`：建立 EventSource 连接，监听 `warning-created` 事件：
    - **level=3（高危）**：弹出红色预警弹窗 + **自动切换至预警中心标签页**（`switchTab('warnings')`）+ 刷新第 1 页预警列表（`loadWarnings(1)`），确保医生第一时间看到预警；
    - **level<3（普通）**：右上角轻提示通知；若当前已在预警中心则静默刷新列表；
  - `warning-updated` 事件：刷新实时统计与列表。
  - `loadRealtimeStats()`：页面初次加载时兜底获取未读/高危/今日新增统计。
  - `markWarningRead(item)`：查看预警详情时自动标记已读，并同步减少未读计数。
  - `goHandleWarningPopup()`：高危弹窗"立即处理"按钮，跳转预警中心并打开处理弹窗。
- `mounted()` 登录成功后调用 `loadRealtimeStats()` + `connectWarningStream()`；`beforeUnmount()` 关闭 SSE 连接、清理重连定时器，防止连接残留。
- 预警详情弹窗（`warning-detail`）新增"已读状态""处理截止时间"展示；新增高危预警实时提醒弹窗（`warning-alert`）。

### `index.html`

- 顶部导航栏铃铛未读角标绑定源由仅 `profile.unreadCount` 改为叠加 `profile.unreadCount + realtimeStats.unread`，有预警未读时点击优先跳转"预警中心"。

## 五、验证方式

### 5.1 基础验证 — 预警列表查询

1. 登录医生账号（`doctor01` / `doctor123`），点击左侧「W 预警中心」，确认列表显示 25 条预置预警数据，可按等级、状态、老人ID 筛选。

### 5.2 实时推送验证 — 上传体征触发预警

1. 确保应用和 Redis 均已启动，登录医生账号（**张 doctor01，ID=2**）。
2. 打开浏览器 F12 控制台，执行：
```js
// 上传陈大爷（elderId=5，属于张医生管理）的收缩压185，触发 systolic>=180 高危规则
fetch('/api/vitals/upload', {
  method: 'POST',
  headers: { 'Authorization': 'Bearer ' + localStorage.getItem('token'), 'Content-Type': 'application/json' },
  body: JSON.stringify([{ elderId: 5, dataType: 1, dataValue: 185, unit: 'mmHg' }])
}).then(r => r.json()).then(console.log)
```
3. 预期结果：
   - ✅ **页面自动从工作台跳转到预警中心**（`switchTab('warnings')`）
   - ✅ **弹出红色高危预警弹窗**，标题为「⚠ 收缩压超过180mmHg，紧急处理！」
   - ✅ 弹窗显示老人ID、预警值、预警内容，可点击「知道了」或「立即处理」
   - ✅ 预警中心列表第一条即为新生成的红色高危预警

### 5.3 事件日志验证

1. 调用 `PUT /api/warnings/{id}/handle`、`/ignore`、`/read`。
2. 检查 `warning_event_log` 表是否生成对应事件记录（event_type: 4处理/5忽略/3已读）。

### 5.4 断线重连验证

1. 手动停止 Redis 或重启后端模拟连接中断。
2. 确认前端提示"连接断开"并在约 5 秒内自动重连成功。

### 5.5 注意事项

- **医生匹配**：SSE 推送按老人档案中 `doctor_id` → `sys_user.id` 匹配，上传体征时需选择当前登录医生负责的老人（如张医生负责的 `elderId IN (5,6,7,14,17)`）。
- **单次执行**：避免连续多次调用上传接口产生大量重复预警，已在数据中预置 25 条测试记录。
- **血氧规则**：`warning_rule` 表中 `bloodOxygen<=90` 规则的 `metric_code` 与上传接口 `getMetricCode(6)="spo2"` 不匹配，暂不能通过上传触发，可手动新建预警测试。

## 七、集成测试问题与修复记录（2026-07-06）

### Issue #1：`health_warning` 表缺少新增字段导致 API 500

- **现象**：`GET /api/warnings` 返回 `500 系统内部错误`，日志显示 `Unknown column 'source_data_id' in 'field list'`
- **根因**：`schema.sql` 中使用 `DELIMITER $$ / CREATE PROCEDURE / CALL` 方式添加字段，但 Spring Boot 的 `ScriptUtils` 不支持 `DELIMITER` 语法，导致存储过程创建失败，四个新字段从未被添加到表中
- **修复**：手动执行 `ALTER TABLE` 语句补全字段；文档更新已知问题说明，建议后续改为 `ALTER TABLE ... ADD COLUMN IF NOT EXISTS`

### Issue #2：Redis pub/sub 监听器 `NullPointerException`

- **现象**：日志反复出现 `MessageListenerAdapter: Listener execution failed`，`this.invoker is null`
- **根因**：自定义 `RedisMessageListenerContainer` Bean 与 Spring Boot 自动配置的容器冲突，导致消息到达时无法找到正确的代理方法
- **修复**：
  - `RedisMessageConfig` 添加 `@ConditionalOnProperty(name="medical.warning.redis-pubsub.enabled", matchIfMissing=false)`，默认禁用 Redis pub/sub
  - `WarningServiceImpl.publish()` 增加 `warningSseManager.pushToUser()` 本地直接推送逻辑，单机模式下不再依赖 Redis 消息通道

### Issue #3：高危预警弹出后未自动跳转预警中心

- **现象**：高危预警弹窗出现后仍停留在原页面，用户需手动点击侧边栏进入
- **修复**：`connectWarningStream()` 的 `warning-created` 事件处理中，level=3 时增加 `this.switchTab('warnings')` + `this.loadWarnings(1)` 自动导航

### Issue #4：血氧规则 `metric_code` 不匹配

- **现象**：`warning_rule` 表中 `bloodOxygen<=90` 规则无法通过 `POST /api/vitals/upload` 触发
- **根因**：`VitalSignServiceImpl.getMetricCode(6)` 返回 `"spo2"`，而规则的 `metric_code` 为 `"bloodOxygen"`
- **状态**：已记录，暂未修复（建议统一为 `bloodOxygen` 或按需添加 `spo2` 规则）
