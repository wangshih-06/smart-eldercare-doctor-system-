# 智慧医养大数据公共服务平台医生服务系统 — 项目后续优化建议

> 适用版本：v1.0.0 （Spring Boot 2.7.18 + MyBatis-Plus + MySQL + JWT）
> 编写日期：2026-06-30

本文档基于当前项目结构（Controller 13 个、Service 10 个、Entity 19 个、Mapper 19 个）梳理后续可优化的方向，按 **优先级** 划分，便于排期推进。

---

## 一、安全与权限（P0 高优先级）

### 1. 配置项与敏感信息外置
- `application.yml` 中数据库密码（`958888`）、Druid 监控账号（`admin/123456`）、JWT 密钥均为明文硬编码，存在泄露风险。
- **建议**：
  - 使用环境变量或 `application-{profile}.yml` 分离 dev / test / prod 配置。
  - 接入 Jasypt 对密码字段加密，或采用 Nacos / Apollo / Vault 等配置中心。
  - JWT `secret` 通过环境变量注入，长度建议 ≥ 64 字节并定期轮换。

### 2. 密码存储统一化
- `AuthServiceImpl` 同时兼容明文与 BCrypt（`storedPassword.startsWith("$2a$")` 判断分支），属过渡逻辑。
- **建议**：
  - 编写一次性迁移脚本，将所有明文密码转为 BCrypt（cost ≥ 10）。
  - 迁移完成后删除明文分支，强制 BCrypt 校验。
  - 引入密码复杂度校验（长度、字符种类、与历史密码比对）。

### 3. 鉴权与防护增强
- 当前仅有 `JwtInterceptor`，且 Token 同时支持从 query parameter 获取（`request.getParameter("token")`），易在日志、Referer 中泄露。
- **建议**：
  - 仅允许从 `Authorization` 头读取 Token，删除 query parameter 回退。
  - 引入 Refresh Token + 短期 Access Token 双 Token 机制。
  - 接入 Spring Security 或 Sa-Token，统一管理 RBAC 角色与接口级权限（医生 / 管理员 / 护士等）。
  - 登录接口增加失败次数限制（IP + 账号维度）、滑动验证码、设备指纹。
  - Druid 监控页 `/druid/*` 必须在生产环境禁用或强口令 + 内网限制。

### 4. 接口防护
- 缺少 XSS、SQL 注入、CSRF、防重放等通用防护。
- **建议**：
  - 加入 `XssFilter`、参数级 `@Validated`、`Hibernate Validator` 全量校验。
  - 重要写操作（处方、转诊、预警）增加幂等性 Token。
  - 接入限流（Sentinel / Resilience4j），按接口、IP、用户限速。

### 5. 数据合规
- 涉及老年人健康档案、病史、用药等敏感个人信息，需符合《个人信息保护法》《数据安全法》。
- **建议**：
  - 身份证号、手机号、住址在 DB 落库前进行 AES 加密或脱敏存储。
  - 列表查询统一返回脱敏后字段，详情接口才返回完整数据。
  - 全链路操作日志（`SysOperationLog` 已有雏形）补充：请求参数、响应、操作前后快照，并归档至 ELK / OSS。

## 二、架构与代码质量（P1）

### 1. 引入 Redis 缓存
- 当前 `pom.xml` 已注释 Redis 依赖。生产环境必备：
  - JWT 黑名单 / 主动登出
  - 验证码（`CaptchaController`）存储
  - 热点数据缓存：医生信息、老人档案、字典、预警规则
  - 分布式锁（处方提交、转诊单生成等关键操作）
- **建议**：启用 Redis，并将 Caffeine 作为本地一级缓存。

### 2. 多模块拆分
- 当前单体单模块结构。随着实体增多（19 个），后续可演化为：
  ```
  medical-doctor-parent
    ├── medical-doctor-common      // 公共工具、注解、异常
    ├── medical-doctor-entity      // DO / DTO / VO
    ├── medical-doctor-mapper      // 数据访问层
    ├── medical-doctor-service     // 业务层
    └── medical-doctor-api         // Controller / 启动类
  ```
- 后续若衍生患者端、家属端、可穿戴接入端，可平滑过渡到微服务（Spring Cloud Alibaba）。

### 3. DTO / VO 分层
- 当前 Controller 直接返回 Entity（如 `AuthService.getUserInfo` 把 `password` 置 null 后返回 `SysUser`）。
- **建议**：
  - 输入用 DTO（带校验注解），输出用 VO，避免敏感字段泄露。
  - 引入 MapStruct 做对象映射，减少手写 setter。

### 4. 通用响应与异常体系
- `R.java`、`BusinessException`、`GlobalExceptionHandler` 已具雏形。可继续完善：
  - 错误码统一编排（模块前缀 + 业务码），形成 `error-code.md` 字典。
  - 国际化（i18n）支持，便于多语言扩展。
  - `GlobalExceptionHandler` 区分 `MethodArgumentNotValidException`、`ConstraintViolationException`、`BindException` 等更细类型。

### 5. 异步与事件解耦
- 预警生成、登录日志、操作日志当前同步写入，影响主流程性能。
- **建议**：
  - 启用 `@Async` + 自定义线程池，日志、消息推送异步处理。
  - 引入领域事件（Spring `ApplicationEventPublisher`）或 MQ（RocketMQ / Kafka），解耦健康预警、随访提醒、家属通知等场景。

### 6. 事务与并发
- 检查所有 `ServiceImpl` 的写方法是否标注 `@Transactional(rollbackFor = Exception.class)`。
- 涉及多表写入的接口（如转诊、干预记录）务必显式声明事务边界，并考虑乐观锁字段 `version`。

## 三、数据库与持久层（P1）

### 1. 启动期建库脚本
- `spring.sql.init.mode: always` 每次启动都执行 schema/data，开发友好但生产风险高。
- **建议**：
  - 生产改为 `never`，由 Flyway / Liquibase 管理版本化迁移。
  - 历史 `schema.sql`、`data.sql`、`init_schema.sql` 三份脚本合并成有版本号的迁移文件链。

### 2. 索引与慢 SQL 治理
- 健康监测、预警、随访属高频读写表。
- **建议**：
  - 为 `elder_id`、`create_time`、`status` 等高频 where / order 字段建联合索引。
  - 接入慢 SQL 日志（`slow_query_log`）与 `p6spy`，定期扫描。
  - 体征数据（`VitalSignData`）量级大，按月分表或迁移至 TDengine / InfluxDB 等时序库。

### 3. 逻辑删除策略
- 已配置 `logic-delete-field: deleted`，注意：
  - 唯一索引需联合 `deleted` 字段，避免逻辑删除后无法重新创建同名记录。
  - 报表统计需明确"是否包含已删除"。

### 4. 大字段 / JSON 字段
- 病史描述、随访记录文本较长，建议：
  - 使用 `MEDIUMTEXT` 或独立明细表，避免行长度膨胀。
  - 富文本入库前做 HTML 白名单过滤（防 XSS）。

## 四、可观测性与运维（P1）

### 1. 日志体系
- 当前 `logging.level.com.medical: debug` 与 MyBatis 控制台输出仅适用开发。
- **建议**：
  - 引入 Logback + 异步 Appender，按日期 / 大小切分。
  - 接入 ELK / Loki + Grafana，关键链路使用 MDC 注入 `traceId`。
  - 业务侧关键操作输出结构化 JSON 日志。

### 2. 监控指标
- 引入 Spring Boot Actuator + Micrometer + Prometheus。
- 关键看板：JVM、连接池、QPS、慢接口、预警触发频次、随访完成率。

### 3. 链路追踪
- 接入 SkyWalking 或 OpenTelemetry，便于后续微服务化和性能定位。

### 4. 健康检查与优雅停机
- 暴露 `/actuator/health`、`/actuator/info`，对接 K8s liveness / readiness。
- 配置 `server.shutdown=graceful`，避免请求中断。

## 五、接口与文档（P2）

### 1. API 规范
- 已使用 Knife4j 3.0.3（Swagger 2 时代版本），建议升级到 Knife4j 4.x（OpenAPI 3）以匹配 Spring Boot 3 升级路径。
- 所有 Controller 补齐 `@Operation`、`@Parameter`、错误码示例。
- 引入 `springdoc-openapi`，自动生成离线文档。

### 2. 版本化
- URL 引入 `/api/v1/...` 前缀，为后续不兼容升级留接口。

### 3. 文件上传
- `file.upload-path: ./upload` 直接落本地磁盘，集群部署会失效。
- **建议**：抽象 `FileStorageService`，可切换本地、MinIO、阿里 OSS、腾讯 COS。

### 4. EasyExcel 导出
- 大数据量导出应改为流式 + 分页 + 异步任务（生成下载链接），避免 OOM。

## 七、测试与质量保障（P2）

1. **单元测试**：`src/test` 目前缺失（项目仅含 spring-boot-starter-test 依赖）。建议：
   - Service 层覆盖率目标 ≥ 60%，使用 Mockito + H2 内存库。
   - Mapper 层用 MyBatis 真实 SQL + Testcontainers MySQL。
2. **集成测试**：通过 `MockMvc` / `RestAssured` 对 Controller 做端到端验证。
3. **契约测试**：若后续多端协同，引入 Pact 保障接口契约。
4. **静态检查**：接入 SonarQube / SpotBugs / Checkstyle + Git Hook（pre-commit）。
5. **CI/CD**：GitLab CI / GitHub Actions / Jenkins 流水线，至少包含编译、单测、镜像构建、推送。

---

## 八、部署与基础设施（P2）

1. **容器化**：编写 `Dockerfile`（多阶段构建）+ `docker-compose.yml`（MySQL + Redis + 应用），降低环境差异。
2. **K8s 部署**：Helm Chart 化部署，配置 HPA、PDB、Liveness/Readiness。
3. **配置分离**：dev / test / staging / prod 独立配置，敏感项接入 KMS。
4. **数据库高可用**：MySQL 主从 + ProxySQL / MHA，或迁移到 PolarDB / RDS。
5. **备份与灾备**：每日全量 + binlog 增量备份；异地容灾。

## 附：风险清单速览

| 风险 | 等级 | 影响 |
| --- | --- | --- |
| 数据库密码/JWT 密钥明文 | 高 | 一旦泄露可直接攻陷系统 |
| 明文密码登录分支 | 高 | 撞库 / 拖库后用户密码裸奔 |
| `sql.init.mode=always` | 中 | 误操作可能重置生产数据 |
| Token 支持 query 参数 | 中 | URL 日志、Referer 泄露 |
| 缺少限流/防重放 | 中 | 易被刷接口、DoS |
| 单体单库无备份策略 | 中 | 故障恢复时间长 |
| 缺少单元测试 | 中 | 迭代风险随代码量上升 |
| 上传仅本地磁盘 | 低 | 集群部署阻塞 |
