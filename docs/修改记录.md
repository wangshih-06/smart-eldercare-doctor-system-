## [2026-07-06] 步骤 D1：随访计划 completed_count 与实际记录数同步

**类型**：数据修复
**模块**：随访管理 / 数据库

### 问题描述
- 现象：随访计划列表中"已完成次数"（`completed_count`）与"查看历史记录"中实际记录数严重不匹配
  - 例如：王大爷高血压随访显示 15/24，但实际只有 8 条记录
  - 例如：孙大爷高血压随访显示 19/24，但实际只有 1 条记录
- 影响：用户看到的进度条虚假，与实际工作不符，无法正确评估随访完成情况

### 根因
种子数据（`sql/c1_seed_follow_records.sql`、`sql/add_more_follow_records.sql`、`db/data.sql`）通过 SQL 直接 INSERT 写入 `follow_record` 表，**绕过了** `FollowUpServiceImpl.createRecord()` 中自动递增 `completed_count` 的逻辑。导致 `completed_count` 是手工填入的任意值，与 `follow_record` 实际行数不匹配。

### 修改内容
- 数据库：`medical_doctor.follow_plan`
  - 改动：执行 UPDATE 语句，以 `follow_record` 实际记录数为准，同步所有计划的 `completed_count` 和 `status`
  - SQL：
    ```sql
    UPDATE follow_plan p
    SET p.completed_count = (SELECT COUNT(*) FROM follow_record r WHERE r.plan_id = p.id),
        p.status = CASE 
            WHEN (SELECT COUNT(*) FROM follow_record r WHERE r.plan_id = p.id) >= p.total_count THEN 2
            ELSE 1
        END;
    ```

### 修复结果
| 计划 | 修复前 completed | 修复后 | 实际记录 |
|------|:---:|:---:|:---:|
| 王大爷高血压 | 15 | 8 | 8 |
| 李奶奶糖尿病 | 6 | 8 | 8 → status=已完成 |
| 孙大爷高血压 | 19 | 1 | 1 |
| …其余 13 个计划 | 各有偏差 | 全部对齐 | — |

### 验证方式
- [x] MySQL 查询 `SELECT p.completed_count, COUNT(r.id) FROM follow_plan p LEFT JOIN follow_record r ON p.id=r.plan_id GROUP BY p.id` → 全部一致
- [x] 李奶奶糖尿病随访（planId=2）status 自动变为 2（已完成），因其 8/8 全部完成
- [x] 其余计划 status=1（进行中）

### 经验教训
- **所有种子数据脚本必须通过 Service 层写入**，不能绕过业务逻辑直接 INSERT
- 后续版本应在 `follow_plan` 表加触发器或定期巡检任务，自动修正 `completed_count` 偏差
- 或者改用视图 `COUNT(*)` 动态计算而非冗余存储（性能 trade-off）
