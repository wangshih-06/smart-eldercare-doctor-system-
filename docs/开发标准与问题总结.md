## 一、开发标准（3 条铁律）

### 1️⃣ 仅修明显 Bug，性能优化暂不处理
- ✅ 修复**直接影响功能正常运行**的 Bug
- ❌ 不做性能优化、重构、技术债清理
- 📋 性能问题统一入档 [问题与优化总览.md](file:///c:/Users/30816/Desktop/智慧医养医生服务系统1.0版本/docs/问题与优化总览.md) §六

### 2️⃣ 单一聚焦，控制合并频率和范围
- 每次修改**聚焦一个**功能或 Bug
- 不在同一次提交里"顺手优化"其他模块
- 避免 PR 过大导致 Code Review 困难

### 3️⃣ 最小化团队整体工作量
- 不做过度设计
- 不写"以后可能用到"的代码
- 复用现有接口、组件、样式（不重复造轮子）

## 四、关键经验（5 条铁律）

### 1️⃣ 环境问题占 90%，先看环境再改代码
- Redis 没启 → 验证码 500
- MySQL 密码错 → Druid 初始化失败
- Spring Boot 没启 → 浏览器看到其他应用

### 2️⃣ 前后端双重校验，多角色操作不能信前端
- 转诊协同：前端 v-if 显示按钮 + 后端状态机 + 后端所有权校验
- 三层缺一不可

### 3️⃣ 静态资源必须带版本号
- `<script src="./app.js?v=时间戳">`
- 改完告诉用户"**Ctrl + F5**"，不要说"刷新一下"

### 4️⃣ 前端文件改完同步 target/classes
- Spring Boot 读 classpath，不读 src
- IDE 增量编译对静态资源不友好
- 后续用 spring-boot-devtools 自动热部署

### 5️⃣ 数据库变更必须写迁移脚本
- ALTER TABLE 单独成文件 `sql/b3_xxx.sql`
- 不能用 ORM 自动迁移（团队可能没装 ORM）
- CI 加 schema 一致性检查
